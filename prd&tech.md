# 专业英语文章精读 App

> 面向出入职场人群的「专业内容驱动型」英语精读产品

---

## 一、PRD（产品需求文档）

### 1. 产品背景与目标

#### 1.1 背景

大量出入职场用户需要**通过英文获取专业信息**（金融、经济、社会学、互联网技术等），但在实际阅读中面临以下问题：

- 专业文章信息密度高，容易在前几段放弃
- 即使逐句翻译，也难以理解作者的**核心论证与立场**
- 学到的语言无法迁移到工作中的表达（汇报、讨论、写作）

现有英语学习类 App 更偏向语言训练（词汇 / 听力 / 语法），而专业阅读工具又默认用户具备较高英语能力，二者之间存在明显断层。

#### 1.2 产品目标

本产品的目标不是“教英语”，而是：

> **帮助用户在有限时间内真正读懂一篇专业英文文章，并沉淀可复用的专业表达方式。**

衡量成功的核心标准：
- 用户是否完成了一次“原本会放弃的专业阅读”
- 用户是否能够用自己的话复述文章核心观点

---

### 2. 目标用户与使用场景

#### 2.1 目标用户

- 出入职场 0–5 年
- 英语水平约 A2–B2（能读，但读得慢、读不深）
- 职业背景包括但不限于：
  - 金融 / 投研 / 咨询
  - 互联网 / 产品 / 技术
  - 社会科学 / 研究型岗位

#### 2.2 核心使用场景

- 每天碎片时间（10–15 分钟）
- 希望通过一篇高质量英文文章获取专业输入
- 对“泛读”和“刷资讯”感到焦虑，希望更高投入产出比

---

### 3. 产品核心定位（一句话）

> **用结构化精读，降低专业英文文章的理解门槛。**

---

### 4. 核心使用流程（每日精读 Flow）

```
进入 App
 → 今日专业文章
 → 阅读地图（专业背景 + 论证结构）
 → 关键段落精读（约 30–40% 内容）
 → 专业表达与论证语言拆解
 → 理解检查（专业问题）
 → 专业复述 / 输出
 → 今日沉淀总结
```

总时长控制在 **10–15 分钟**。

---

### 5. 功能模块拆解

#### 5.1 今日文章模块

**文章要求：**
- 专业领域明确（金融 / 经济 / 社会 / 技术）
- 300–600 词
- 观点清晰、论证完整

**文章元数据：**
- 领域
- 专业难度
- 阅读建议人群
- 文章类型（解释 / 观点 / 反驳）

---

#### 5.2 阅读地图（Pre-read）

在正文前提供：
- 核心问题（作者想解决什么）
- 主要结论
- 论证结构概览（文字版）

目的：**先给认知框架，再进入语言细节**。

---

#### 5.3 关键段落精读

- 只精读 3–5 个关键段落
- 每个段落包含：
  - 段落功能说明（定义 / 推导 / 反驳）
  - 关键句标注

---

#### 5.4 专业语言与表达拆解

重点讲解：
- 用于论证的句型
- 不确定性、假设、因果关系表达
- 专业语境下的审慎措辞

避免：
- 全文翻译
- 与理解无关的语法扩展

---

#### 5.5 理解检查

形式：
- 开放式问题
- 可中英混合回答（初期）

示例：
- 作者的核心假设是什么？
- 哪个因素对结论影响最大？

---

#### 5.6 输出（Professional Output）

输出目标：**专业复述，而非自由创作**。

形式示例：
- 用 2 句话向同事解释文章结论
- 用更谨慎的语气改写一个结论

AI 提供反馈：
- 逻辑是否清楚
- 是否符合专业语境
- 哪些表达可优化

---

#### 5.7 今日沉淀总结

每天只沉淀三项内容：
1. 1 个专业论证句型
2. 1 个关键概念或逻辑结构
3. 1 个可复用观点表达

---

### 6. MVP 范围定义

#### 6.1 MVP 必须包含
- 单一领域（如 AI / 宏观经济）
- 每日 1 篇文章
- 阅读地图
- 关键段落精读
- 1 次专业输出 + AI 反馈

#### 6.2 MVP 明确不做
- 社交系统
- 排行榜 / 打卡
- 多领域切换
- 复杂订阅体系

---

## 二、技术设计文档

### 1. 系统整体架构

```
客户端（Web / App）
        ↓
API 层（FastAPI）
        ↓
业务服务层
  ├─ 内容管理
  ├─ 精读生成
  ├─ 用户输出评估
        ↓
LLM Agent 层
        ↓
数据库 / 向量库
```

---

### 2. 核心数据模型（简化）

#### 2.1 Article
- id
- title
- domain
- difficulty
- content
- metadata

#### 2.2 Paragraph
- id
- article_id
- text
- role（定义 / 论证 / 反驳）

#### 2.3 SentenceFocus
- id
- paragraph_id
- sentence
- explanation
- transferable_expression

#### 2.4 UserOutput
- id
- user_id
- article_id
- content
- feedback

#### 2.5 MemoryItem
- id
- user_id
- type（句型 / 概念 / 表达）
- content

---

### 3. Agent 设计

#### 3.1 Domain Curator Agent
- 识别文章领域
- 判断专业密度
- 推荐精读比例

#### 3.2 Argument Mapper Agent
- 抽取核心问题
- 识别假设与结论
- 输出论证结构

#### 3.3 Language-for-Reasoning Agent
- 只解释用于理解论证的语言
- 输出可迁移表达

#### 3.4 Professional Feedback Agent
- 评估用户输出的专业性
- 给出逻辑与措辞建议

---

### 4. LLM 使用原则

- 分 Agent 调用，避免“万能 Prompt”
- 所有解释围绕：**是否帮助理解专业内容**
- 用户输出反馈以“建议”为主，不给标准答案

---

### 5. 技术选型建议

#### 5.1 后端技术栈

- 运行环境：Node.js（LTS）
- Web 框架：NestJS（推荐）或 Express
- 语言：TypeScript
- 数据库：MongoDB（文档型，适合内容与精读结构）
- ODM：Mongoose

#### 5.2 前端技术栈

- 框架：Flutter
- 目标平台：
  - iOS / Android（主）
  - Web（可选，Flutter Web）

#### 5.3 AI / LLM 接入层

- 独立 AI Service（Node.js Service）
- Agent 以“任务型接口”方式暴露
- 支持异步生成与结果缓存

#### 5.4 基础设施（MVP）

- API 部署：Docker
- 文件存储：对象存储（文章原文、日志）
- 鉴权：JWT

技术选型原则：
- 优先内容生成与体验验证
- 避免过早引入复杂中间件

---

### 6. 后续扩展方向

- 多领域精读模板
- 用户个人专业表达库
- 跨文章观点对比
